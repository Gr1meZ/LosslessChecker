# LosslessChecker — Комплексный аудит и улучшение: Дизайн

Дата: 2025-06-25
Статус: Утверждён

## Обзор

Комплексная доработка LosslessChecker по 4 направлениям:
1. Память и стабильность (предотвращение OOM)
2. Исправление алгоритмов
3. Спектрограмма (оси, cutoff, zoom, MP3-режим)
4. UI (сортировка, фильтрация, экспорт, темы)

## 1. Память и стабильность

### Проблема
`AudioDecoder.Decode` загружает весь PCM в `List<float>` × 2 канала → `.ToArray()`. Для 5-минутного стерео 44.1kHz = ~240 MB float-данных. 8 параллельных воркеров = ~2 GB на пике. Плюс `AudioAnalyzer` создаёт `mono = new float[buffer.Length]` (дубликат всего сигнала). `BitDepthValidator.CheckLsbZeroPaddedFull` создаёт ещё один `float[n]` массив. `ComputePcmMd5` создаёт `byte[samples.Length * 2]` (до 100+ MB).

### Решение: гибридный подход

**Новый класс `AudioPipeline`** — заменяет разрозненный пайплайн в `AudioAnalyzer`:
- Единый mono-буфер `float[]` — выделяется один раз при декодировании, шарят все анализаторы
- Чанковые анализаторы (DR, TruePeak, DC, Phase, BitDepth) читают `ReadOnlySpan<float>` из mono
- FFT-анализаторы (Cutoff, Artifacts, Upscale, Vinyl, Spectrogram) используют тот же mono
- После завершения трека: `mono = null; buffer = null`
- `GC.Collect(0, GCCollectionMode.Optimized)` каждые 10 треков

**`MainViewModel` — ограничение параллелизма:**
- `SemaphoreSlim(2)` — максимум 2 воркера одновременно (вместо `Environment.ProcessorCount`)
- Сохранить `ConcurrentQueue` модель (уже делает backpressure)

**`AudioDecoder` — убрать отладочный вывод и дублирование:**
- Убрать `LogLevel` (пишет `lossless_dr_debug.log` на каждый трек — лишний I/O)
- Убрать `.ToArray()` где не нужно

**`BitDepthValidator` — убрать копию буфера:**
- `CheckLsbZeroPaddedFull`: не создавать `float[n]` mono-массив, работать с `buffer.Left` / `buffer.Right` напрямую
- LSB-проверка без предварительного микширования

**`ContainerAnalyzer.ComputePcmMd5` — инкрементальный хеш:**
- Заменить `new byte[samples.Length * 2]` → `IncrementalHash` с чанками по 4096 семплов

**`SpectrogramBuilder` — переиспользование FFT-буферов:**
- `frame`, `real`, `imag` выделяются один раз, а не на каждый кадр/проход

## 2. Исправление алгоритмов

### 2.1 LUFS K-weight фильтр (критическая)

**Проблема:** Коэффициенты `0.99` и `0.2` жёстко зашиты в `LufsMeter.AnalyzeMono`. Они корректны только для 48 kHz.

**Исправление:** Вычислять динамически на основе `sampleRate`:

```csharp
double omegaHp = 2.0 * Math.PI * 38.0 / sampleRate;
double hpCoeff = Math.Exp(-omegaHp);           // ≈ 0.99 для 48k, ≈ 0.989 для 44.1k
double omegaSh = 2.0 * Math.PI * 1500.0 / sampleRate;
double shCoeff = 0.5 * (1.0 - Math.Exp(-omegaSh)); // ≈ 0.2 для 48k, ≈ 0.187 для 44.1k
```

Формулы из ITU-R BS.1770-4 Annex 1.

### 2.2 DR Meter — подсчёт клиппинга

**Проблема:** При `_consecutive >= ClipRunMin (3)` счётчик обнуляется. Последовательность из 9 клипнутых семплов считается как 3 отдельных серии вместо одной.

**Исправление:**
```csharp
if (++_consecutive == ClipRunMin) _clippedRuns++;
// Сбрасывать _consecutive = 0 только когда abs < 1.0f
else if (_consecutive > ClipRunMin) { /* уже учтено */ }
```

### 2.3 MetricsCoverage — вырожденная проверка LUFS

**Проблема:** Условие `r.IntegratedLufs < -7 || r.IntegratedLufs > -1` — условие `> -1` практически всегда истинно для музыки, метрика всегда `passed`.

**Исправление:**
```csharp
if (r.IntegratedLufs <= -7 && r.IntegratedLufs > -70) passed++;
```
LUFS считается пройденным если не экстремально громкий (> -7) и не является значением-заглушкой (-100).

### 2.4 BitDepth LSB — banker's rounding

**Проблема:** `(int)Math.Round(sample * 8388607.0)` использует banker's rounding (к ближайшему чётному), что для `.5` даёт неожиданный результат.

**Исправление:**
```csharp
int sample24 = (int)(sample * 8388607.0 + 0.5 * Math.Sign(sample));
```

### 2.5 ResamplingDetector — сырые FFT-магнитуды

**Проблема:** Детектор анализирует квантованные `byte` (0-255) из спектрограммы, теряя ~48 dB точности.

**Исправление:** Передать в `ResamplingDetector` сырой `double[] spectrum` (averaged FFT magnitudes) и размеры до квантования. Пороги пересчитать в dB-домене.

## 3. Спектрограмма

### 3.1 Логарифмическая частотная шкала

Вместо линейного маппинга 256 бинов FFT → 256 пикселей:
- Log-маппинг: `freq(pixel) = 10^(log10(fMin) + pixel/height * (log10(Nyquist) - log10(fMin)))`
- fMin = 20 Hz (нижняя граница слышимого диапазона)
- Низкие частоты получают пропорционально больше вертикального пространства

### 3.2 Оси и сетка

Отрисовка поверх `WriteableBitmap`:

**Ось Y (частота):**
- Метки на: 20, 50, 100, 200, 500, 1k, 2k, 5k, 10k, 20k Hz (обрезается по Nyquist)
- Подписи слева от спектрограммы, шрифт 9pt, цвет `#585b70`

**Ось X (время):**
- Метки каждые 30 сек для треков < 5 мин, каждую минуту для > 5 мин
- Подписи снизу, тот же стиль

**Сетка:**
- Полупрозрачные линии (`#45475a`, opacity 0.3) на основных делениях
- Пунктирная сетка на промежуточных делениях

**Линия Cutoff:**
- Горизонтальная пунктирная линия (`#f38ba8`, 2px dash) на частоте `SelectedCutoffFrequency`
- Подпись `"Срез: XXXX Гц"` справа от линии

**Цветовая легенда:**
- Вертикальный градиент-бар (16px ширина) справа от спектрограммы
- Метки: 0, -24, -48, -72, -96 dB
- Заголовок `"dB"` сверху

### 3.3 Окно спектрограммы (попап)

Замена текущего `SpectrogramWindow`:

- **Canvas** вместо просто `Image` — позволяет overlay осей и cutoff
- **Zoom:** колёсико мыши — увеличение по X (время) и Y (частота) независимо; `Ctrl+колесо` — только X, `Shift+колесо` — только Y
- **Pan:** перетаскивание левой кнопкой (когда zoom активен)
- **Reset zoom:** двойной клик
- **Клик по cutoff-линии:** выделяет соответствующую метрику в таблице
- **Ресайз окна:** спектрограмма перестраивается с новым разрешением
- **Кнопка «Копировать как PNG»** в тулбаре окна
- **Инфо-панель снизу:** частота и dB под курсором (при наведении)
- **Заголовок окна:** `"Спектрограмма — ИмяФайла"`

### 3.4 MP3-режим

При обнаружении lossy-файла (.mp3, .m4a, .alac с признаками сжатия) — специальный режим оценки:

**MP3 Quality Score (0–100%):**
- Битрейт из заголовка MP3 через NAudio (`Mp3WaveFormat`)
- Фактический частотный срез vs. ожидаемый для битрейта:
  - 128 kbps → ~16 kHz, 192 kbps → ~18 kHz, 256 kbps → ~19 kHz, 320 kbps → ~20 kHz
- Артефакты сжатия: чем меньше pre-echo/spectral holes/sizzle — тем выше качество рипа
- Соответствие заявленного битрейта реальному срезу (транскод из lossy если не совпадает)

**MP3-панель в UI:**
- Битрейт (CBR/VBR, kbps), encoder (LAME/Xing/FhG/etc из заголовка), stereo mode
- Вердикт для MP3: «Хороший рип 320 kbps» / «Средний 192 kbps» / «Плохой 128 kbps» / «Транскод из lossy»

**MP3-спектрограмма:**
- Cutoff-линия текущего файла
- Зона ожидаемого среза для заявленного битрейта (полупрозрачная полоса)
- Цветовое кодирование: зелёный — срез соответствует битрейту, жёлтый — немного ниже, красный — транскод

## 4. UI

### 4.1 Сортировка

Клик по заголовку колонки DataGrid — сортировка ASC/DESC:
- Название (string), Длительность (double), Формат (string), Cutoff (double), DR (double), Подлинность (string), Кач-во (double), Решение (string)
- Индикатор сортировки (▲/▼) в заголовке

### 4.2 Фильтрация

**Строка над DataGrid:**
- `TextBox` с плейсхолдером «Поиск по названию...»
- Ввод фильтрует `FilteredFiles` по `FileName.Contains(search, OrdinalIgnoreCase)`

**Фильтр-чипсы:**
- `✅ KEEP` `⚠ INVESTIGATE` `❌ REPLACE` `💀 MP3` — toggle-кнопки
- Активные чипсы выделены цветом, неактивные — серые
- Комбинация чипсов: `FilteredFiles = Files.Where(выбранные решения)`

**Фильтр по формату:**
- Выпадающий список (ComboBox): «Все форматы», «FLAC», «WAV», «MP3», «M4A/ALAC»

### 4.3 Экспорт

**Кнопка «📊 Экспорт» в тулбаре (рядом с Выбрать/Стоп):**
- **CSV:** все метрики по всем трекам в табличном виде
- **JSON:** полные `AnalysisResult[]` в JSON (для внешней обработки)
- **HTML-отчёт:** цветокодированная таблица с фильтрацией, сохраняется как `.html`

Форматы через `SaveFileDialog` с фильтром.

### 4.4 Тёмная/светлая тема

**Toggle в тулбаре:** кнопка ☀️/🌙

**Реализация:**
- Все цвета в XAML вынесены в `DynamicResource` (не хардкод)
- Два `ResourceDictionary`: `Themes/Dark.xaml` (текущий Catppuccin Mocha), `Themes/Light.xaml` (новый Catppuccin Latte)
- При переключении: `Application.Current.Resources.MergedDictionaries` swap
- Сохранение выбора в `Properties.Settings`

**Семантические кисти (в дополнение к существующим `App.xaml`):**
- `BgPrimaryBrush`, `BgSecondaryBrush`, `BgTertiaryBrush` (фон)
- `FgPrimaryBrush`, `FgSecondaryBrush`, `FgMutedBrush` (текст)
- `BorderBrush`, `AccentBrush`
- Существующие `LosslessGreenBrush`, `SuspiciousAmberBrush`, `FakeRedBrush` сохраняются

### 4.5 Навигация и UX

**Keyboard в DataGrid:**
- Стрелки вверх/вниз — смена выделенного трека, мгновенное обновление детальной панели

**Двойной клик по треку:**
- Открыть окно спектрограммы (быстрее чем через панель)

**Прогресс-бар с информацией:**
- Текст над прогресс-баром: `"Обработка: Artist — Title (15/500)"`
- Или просто `"Обработано 15 из 500"` если теги ещё не прочитаны

**Приветственный экран (пустое состояние):**
- Когда папка не выбрана — показывать:
  - Иконка 🎵 крупно
  - Текст «Перетащите папку с аудио сюда или нажмите Выбрать папку»
  - Пунктирная рамка (drag-and-drop target)

**Drag & Drop:**
- `AllowDrop="True"` на главном окне
- Принимать папки и отдельные файлы (`.mp3`, `.flac`, `.wav`, `.m4a`, `.alac`)
- При дропе — запускать `ScanAndAnalyze`

**Контекстное меню на треке (правый клик):**
- «📂 Открыть папку с файлом» → `Process.Start("explorer", "/select," + path)`
- «📋 Копировать все метрики» → текущий `CopyMetricsCommand`
- «🔬 Показать спектрограмму» → открыть попап

## Критерии приёмки

- 500 FLAC по 30 MB обрабатываются без OOM на машине с 8 GB RAM
- LUFS совпадает с эталонным измерителем (Youlean, ffmpeg loudnorm) в пределах ±0.5 LUFS
- DR совпадает с foobar2000 DR Meter в пределах ±0.5 dB
- Спектрограмма показывает логарифмическую шкалу с читаемыми метками
- Cutoff линия корректно позиционируется для файлов с известным срезом
- Сортировка и фильтрация работают без ошибок
- Переключение тем не вызывает мерцания и сохраняется между запусками
- Drag-and-drop папки запускает анализ
